Day 25 Lab

Simulate SDN & IoT with Microsoft Azure (Beginner)

Objective: Use the Azure Portal to mimic SDN “intent” and an IoT network zone—no VMs or CLI. You’ll create a VNet with two subnets, attach an NSG with simple allow/deny rules for an IoT subnet, and label resources with tags (policy-style grouping).

Estimated time: 20–30 minutes • Cost: $0 (no compute)

🛠 What You Need

  • Azure account (free tier is fine)
  • Access to the Azure Portal

Step 1 — Create a Resource Group

  1. Portal → search Resource groups → Create.
  2. Name: HL-SDN-RG • Region: your closest (e.g., East US).
  3. Review + create → Create.

Step 2 — Create a VNet with Two Subnets

  1. Create a resource → search Virtual Network → Create.
  2. Resource group: HL-SDN-RG • Name: HL-SDN-VNet • Region: same as RG.
  3. IP addresses tab:
    • IPv4 address space: 10.25.0.0/16
    • Edit subnets:
      • PublicSubnet → 10.25.1.0/24
      • IoTSubnet → 10.25.2.0/24
    • (Optional) Enable IPv6 and add 2001:db8:25::/48 with /64 slices to mirror dual-stack.
  4. Review + create → Create.

Step 3 — Attach an NSG to the IoT Subnet (Simple Policy)

This mimics “policy intent” in SDN: allow MQTT to IoT devices, deny old protocols like Telnet.

  1. Create a resource → Network security group → Create.
  2. Name: HL-IoT-NSG • RG: HL-SDN-RG • Region: same.
  3. Open HL-IoT-NSG → Inbound security rules → Add:
    • Name: Allow-MQTT • Priority: 100 • Protocol: TCP • Port: 1883 • Source: VirtualNetwork • Action: Allow.
    • Name: Deny-Telnet • Priority: 120 • Protocol: TCP • Port: 23 • Source: Any • Action: Deny.
  4. Associate to subnet: Subnets → Associate → VNet HL-SDN-VNet → Subnet IoTSubnet.
Concept: The NSG acts like a distributed firewall policy—very SDN-ish. We’re not deploying devices; just defining intent on the fabric.

Step 4 — Tag Resources (Group by “Intent”)

Tags help you target policy or cost views—think of them like SDN metadata.

  1. Open HL-SDN-VNet → Tags → Edit:
    • env=lab, owner=hoodie
  2. Open each subnet → Tags:
    • tier=public on PublicSubnet
    • tier=iot on IoTSubnet

Later, Azure Policy or automation could use these tags to apply rules programmatically (controller-style).

Step 5 — Explore What You Built

  • VNet → Overview: verify address space 10.25.0.0/16.
  • Subnets: confirm PublicSubnet 10.25.1.0/24 and IoTSubnet 10.25.2.0/24, with NSG attached to IoT.
  • NSG → Inbound rules: verify priority order (100 allow MQTT, 120 deny Telnet, then defaults).
  • Tags: confirm VNet/subnet tags are present.

🎯 Stretch (Optional)

  • Add one more NSG rule on IoTSubnet: Allow-HTTPS (TCP/443, priority 110). Explain why it sits between 100 and 120.
  • Create a second VNet HL-Edge-VNet (e.g., 10.26.0.0/16) and add a VNet peering both ways (no VMs required) to visualize fabric stitching.
  • Write a one-line “policy intent” note you’d push with automation, e.g., “Deny Telnet everywhere in env=lab”.

🧠 Quick Ref

  • VNet = software-defined private network boundary.
  • Subnet = segmented slice of the VNet (useful for zones like public vs. IoT).
  • NSG = simple, distributed firewall policy you can attach to subnets/NICs.
  • Tags = metadata that controllers/policy engines can target.

🧹 Clean Up (recommended)

  • Portal → Resource groups → open HL-SDN-RG → Delete resource group → type the name to confirm.

Lab Summary: You modeled SDN intent in Azure by defining network segments, attaching a policy (NSG) for the IoT zone, and tagging resources for future automation—without spinning up any paid resources.
